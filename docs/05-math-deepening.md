## §8. 把 Rice 定理精化成"算术分层"——这才是分流引擎的真正理由

`01 §6` 用 Rice 定理给了一句话天花板:"程序的一切非平凡语义性质都不可判定。" 但这句话太粗——它没说**有多不可判定**。精确的事实是:**语义性质落在算术层级(arithmetical hierarchy)的确定档位上**,而档位**直接决定**了能用哪种近似把它"够着"。

| 性质 | 算术层级 | 可计算性 | 唯一可行的策略 |
|---|---|---|---|
| 给定输入会停机 | **Σ⁰₁**(r.e./递归可枚举) | 半可判定 | **欠近似**:跑/测试/有界模型检查能**确认**,但不能否定 |
| 可达某坏状态(reachability) | **Σ⁰₁** | 半可判定 | **欠近似 bug-finding**(污点命中、符号执行找到真阳) |
| **安全性:永不进入坏状态** | **Π⁰₁**(co-r.e.) | 其否定半可判定 | **过近似 sound**:抽象解释能**证明**它成立 |
| 终止性(对所有输入停机) | **Π⁰₂** | 更高 | 需 ranking function / ITP |
| 两个程序等价 | **Π⁰₂** | 更高 | ITP / 关系证明 |

**核心洞见**:安全性质是 **Π⁰₁(co-r.e.)**——形如"对所有执行路径,坏事不发生"。坏状态集合是 r.e.(Σ⁰₁),安全性是其补,故 Π⁰₁。**正因为它是 Π⁰₁,sound 的过近似(抽象解释)才能证明它**:在一个**包含**真实行为的超集上验证全称命题,若超集都满足,真实子集必满足(不漏报)。反过来,可达性是 Σ⁰₁,**欠近似**(测试/bounded checking)能抓到真 bug 但不能保证没 bug。

> 这把 `01 §6` 的"四档难度表"从经验排序升级成**有递归论根基的精确分类**:formal-atlas"结构/治理性质走 Datalog 过近似(sound),bug 类走欠近似(找真阳)"的引擎分流,**不是工程口味,而是被性质所处的算术层级强制的**。
>
> 锚点:Rice 定理(Rice 1953);安全 = co-r.e. 的标准对应(Alpern–Schneider 1985"Defining Liveness"的安全/活性划分,以及 Manna–Pnueli 的安全-进展层级)。**待联网恢复补精确文献链接。**

## §9. 描述复杂度:Prolog 层"到底能判定哪一类性质"有精确答案

为什么用 Datalog/Prolog 而不是随手写一堆 `if`?`01 §4` 给了语义答案(最小不动点 = 最小 Herbrand 模型)。再深一层是**计算复杂度的硬边界**,来自**描述复杂度(descriptive complexity)**:

- **Datalog 的数据复杂度是 PTIME**(关于事实库大小多项式);分层否定(stratified negation)的 Datalog 仍在 PTIME。
- **Immerman–Vardi 定理**:**最小不动点逻辑 LFP 在有序有限结构上恰好刻画 PTIME**。即"能用 LFP 表达的关系查询" = "能在多项式时间判定的关系查询"。
- **推论(对 formal-atlas 的直接意义)**:把代码抽成一个有限一阶关系结构(事实库)之后,**Prolog/Datalog 层能 sound 判定的性质,正好是该结构上 PTIME-可计算的关系性质**——可达性 `reaches/2`(传递闭包)、死代码、环 `cyclic/1`、影响面 `impact/2` 全在此列。

**这给了"何时停在 Prolog、何时必须升级引擎"一个理论判据,而非拍脑袋**:一旦性质需要**计数**(如"被调用超过 k 次")、**最优化**、或**高阶/无界量化**(如"存在一个使所有路径满足 φ 的输入"),就溢出了 Datalog 的 PTIME 关系片段,必须上 SMT(§11 的 NP/可判定理论)或 ITP。

> 锚点:Immerman 1986、Vardi 1982(LFP = PTIME on ordered structures);Datalog PTIME 数据复杂度见 Abiteboul–Hull–Vianu《Foundations of Databases》第 12–15 章。**待联网恢复补链接。**

## §10. λ-立方体:把每个验证工具钉在同一张类型论地图上

`01 §1` 讲了 Curry–Howard(命题即类型、证明即程序)。再深一层是 **Barendregt 的 λ-立方体(1991)**——它把类型系统的表达力摆成一个三维设计空间,三条轴分别是允许哪种"依赖":

```
                       λω ───────────── λC = 构造演算 (CIC)
                      /│                /│      ← Coq / Lean / Agda 内核
                     / │               / │        证明 = 程序,表达力天花板
                  λ2 ──┼──────── λP2  │
   (System F,多态)│  │            │  │
                   │ λ→ ──────────┼ λP   (λΠ,依赖类型:类型可依赖值)
                   │/  (STLC)      │/
   Curry-Howard 最小模型 ─────────┘

   轴1 项依赖项 = 普通函数        轴2 项依赖类型 = 多态(泛型)
   轴3 类型依赖项 = 依赖类型(⇒ 类型里能写任意命题,这是通往全证明的门)
```

把现实里的验证工具钉上去:

| 工具 | 在地图上的位置 | 自动化程度 |
|---|---|---|
| 普通带泛型的语言(TS/Java 泛型) | λ2(System F 附近) | 类型检查全自动,但表达力弱 |
| **Dafny / Verus** | **不在立方体顶点**:一阶逻辑 + 外部 **SMT** 放电 | 高自动(SMT),但需写 spec/不变式 |
| **精化类型(§11)** | 依赖类型的**可判定角落**:λP 的受限片段 | **全自动**(SMT 判定),表达力适中 |
| **Lean / Coq / Agda** | **λC 顶点**(CIC):证明 = 程序,表达力最强 | 人工/半自动(需写证明) |

**formal-atlas 现状**:覆盖"结构(Datalog,§9)+ 契约(SMT 骨架)";λC 顶点是 `01 §6` 表里最右档(Logos Research 的主战场,见 `03-atlas-comparison.md §四`)。**地图中间空着一档——见 §11。**

> 锚点:Barendregt 1991"Lambda calculi with types";Coq 内核 = CIC(Calculus of Inductive Constructions,λC 的归纳扩展)。

## §11. 缺失的一档:精化类型(Refinement Types)= Datalog 与 ITP 之间"可判定的功能正确性"

formal-atlas 当前从 **Datalog(可判定结构,§9)** 一步跳到 **"contract/3 → SMT/Dafny 骨架"**,再远眺 **ITP(λC 顶点)**。中间其实有一档**数学上极合身、且能自动化**的台阶被跳过了:**精化类型**

```
{ v : T | φ(v) }      —— "类型 T 中所有满足谓词 φ 的值"
```

关键在于 **φ 取自一个可判定的 SMT 理论**(线性算术、未解释函数、数组…)。于是:

- **Liquid Types**(Rondon–Kawaguchi–Jhala, PLDI 2008):把 φ 限定在 QF-UFLIA 等可判定片段 ⇒ **类型检查可判定**;更妙的是**不变式可自动推断**——其推断算法本质就是**谓词抽象上的抽象解释**:在一个有限的 qualifier 集合上求最小不动点。**这与 `01 §3` 的不动点理论同源**——精化类型推断就是抽象解释的一个实例。
- **F***、**Liquid Haskell**:工业级实现;函数验证 = 让 SMT 判定一串蕴含 `φ_pre ⇒ φ_callee_pre`、`φ_callee_post ⇒ φ_post`。这其实就是**用类型系统写的 Hoare 逻辑**(§与 `01 §2` 的公理语义对应)。

**为什么这一档对 formal-atlas 是天作之合**:

1. **复用已集成的 z3**:`src/verify/smt-bridge.js` + `smt-dsl.js` 已经能把契约 DSL 编译成 z3 项并判定蕴含——精化类型检查就是**同一个判定形态的系统化版本**。
2. **把契约从"自然语言、需人复核"升级为"机器可判定"**:现在 `contract(R, pre, '...')` 是 NL 字符串(§12 的忠实度难题);精化谓词 `refinement(R, arg_i, 'arg_i > 0')` 是**可形式化、可判定**的,落在 `01 §6` 最难需人复核档之外。
3. **恰好落在 §8 的 Π⁰₁ 安全区的可判定子片段**:sound 且自动,不漏报。

> 结论:**精化类型是 formal-atlas 性价比最高的下一档能力**(见 `06-frontier-map.md` ★2)——它把"可判定结构"与"全证明 ITP"之间的鸿沟,用一个**复用现有 z3、SMT-可判定、可自动推断**的台阶补上。
>
> 锚点:Rondon, Kawaguchi, Jhala, "Liquid Types", PLDI 2008;Freeman–Pfenning 1991(refinement types 起源);Vazou et al.(Liquid Haskell);Swamy et al.(F*)。

## §12. 抽象解释的"完备性":近似什么时候不丢精度

`01 §3` 只讲了 **soundness**(过近似不漏报)。但 sound 的抽象可能**误报满天飞**。再深一层是 **Giacobazzi–Ranzato(JACM 2000)的完备性理论**:

- 给定要证的性质 + 一个抽象域,分析可能是 **complete(完备)** 的——意思是**抽象不引入任何精度损失**:在抽象上分析既 sound 又**不产生该性质的误报**。
- 更强的结论:对任意抽象域,存在一个**唯一的最小"完备化(complete shell)"**——可以**系统地把抽象补全**到对目标性质无损,而不必盲目堆细节。

**对 formal-atlas 的可操作意义**:当某条规则(如某个 `violation/2` 或死代码判定)误报偏多时,完备性理论说这不是"再打个补丁"的问题,而是**"当前抽象对这条性质不完备"**——可以有理论指导地**精化抽象**(加哪个谓词、提哪一维),把误报系统性消掉。这正是 formal-atlas 把死代码误报从 86→1 走过的路(points-to + 作用域解析 = 给抽象补维度)的**理论解释**。

> 锚点:Giacobazzi, Ranzato, Scozzari, "Making abstract interpretations complete", JACM 47(2), 2000;Ranzato 后续 completeness/incompleteness 系列。

## §13. 神经符号"忠实度"的数学陈述——整套设想最深的开放问题

把 LLM 那一步讲到底。符号侧的抽象 α(EXTRACT/LINK)对其声明的抽象是 **sound** 的(§8–§9 有递归论与复杂度背书)。但 **AI lifter / formalize 产出的是一个不可靠的近似 α̃**(意图、契约、精化谓词)。这里要极其诚实:

- **generate-and-check 保证的是什么**:`ai-lifter.js` 用 `FACT_LINE` 正则把关语法,下游 Prolog/z3 对"**给定这些事实**"的推理是 sound 的。但这**只是条件 soundness**——结论形如"*若* α̃ 产出的事实为真,*则* 性质成立"。它**不保证 α̃ 的事实本身忠实于代码/意图**。这是整个神经符号管线唯一的 soundness 缺口,必须在文档里讲穿,**绝不能把 LLM 的输出当成定论**。

- **可执行的把关手段(都有数学形态)**:
  1. **回译 round-trip**:`代码 --α̃--> spec --(back-translate)--> 代码'`,检查 `代码 ≈ 代码'`(语义保持)。这是"忠实"的一个**必要条件**(非充分)。
  2. **性质测试 = Popperian 证伪**:把 spec/精化谓词当可执行谓词,喂已知合法/非法样例。**拒绝了已知合法输入,或接受了已知非法输入 ⇒ 该 spec 被证伪**。这就是 `04-roadmap Phase 3`"spec faithfulness"(对标 Verus-SpecGym)的数学内核:**忠实度无法被证明,但能被证伪**——这正是 §8 的 Σ⁰₁/Π⁰₁ 不对称在 LLM 侧的回声。
  3. **多裁判 / 反例驱动**:UNSAT 或 violation → 结构化反馈 → LLM 修订 → 再判定(`06-frontier-map.md` ★3 的闭环)。

- **本质边界(为什么这问题"最深")**:"**intent(意图)**"不在数学之内——它是规约的**来源**而非定理。不存在一个判定"这段自然语言契约是否忠实表达了人类意图"的 oracle,因为意图没有形式定义。**数学能做到的极限**是:"与可观察行为一致(回译 + 测试不被证伪)+ 下游推理 sound"。再往上,是人的复核。

> 这一节是 formal-atlas 区别于"把 LLM 输出直接当真"的工具的**诚实底线**,也是它与 `01 §5` neuro-symbolic 设计承诺("LLM 只产事实、永远过求解器才成结论")的数学展开。
>
> 锚点:Verus-SpecGym / Verus-SpecBench(spec faithfulness 评测);Popper 证伪主义(可证伪 = 可被反例推翻);autoformalization 综述见 `references.md §E`。

## §14. 地平线:同伦类型论(HoTT / 单值基础)——"代码 = 数学"的当代最深表述

你想"从数学角度想透"代码与数学,当代最深的坐标是 **Voevodsky 的单值基础(Univalent Foundations)/ 同伦类型论(HoTT)**:

- **类型 = 空间(∞-群胚)**;**项 = 空间里的点**;**`a = b`(相等)= 从 a 到 b 的路径空间**;高阶相等 = 路径之间的同伦。
- **单值公理(Univalence)**:`(A ≃ B) ≃ (A = B)`——**等价的类型,可以被当作相等**。这把"代码里的同构"提升为"数学上的相等"。

它把 Curry–Howard(`01 §1` 命题即类型)推到极致:**逻辑、计算、拓扑/同伦三位一体**。对 formal-atlas 它**暂不可落地**——过近似 / Datalog / SMT 都不需要这层结构;但它是"程序/类型的数学本性究竟是什么"的**终极答案候选**,也是 Lean/Agda 基础之争的核心。**列为地平线,标记"代码=数学"这句话能被推到多远。**

> 锚点:The Univalent Foundations Program, *Homotopy Type Theory: Univalent Foundations of Mathematics*, 2013(HoTT Book);Voevodsky。
